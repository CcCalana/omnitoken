# T-013 audit_logs schema + admin write middleware PROPOSAL

## 1. 字段定义与类型

当前 `000001_init` 已有占位版 `audit_logs`（`actor` / `ip_address` / `before_state` / `after_state`），但不满足 T-013 的 `actor_id` / `actor_type` / `request_id` 要求。建议新增 `000007_admin_audit_logs.up/down.sql` 演进现表，不回改历史 migration。

推荐目标 schema：

| 字段 | PG 类型 | 说明 |
| --- | --- | --- |
| `id` | `uuid PRIMARY KEY DEFAULT gen_random_uuid()` | 与现有表族一致 |
| `actor_id` | `text NOT NULL` | bootstrap 阶段固定 `bootstrap`；T-005b 后写真实 user UUID 字符串 |
| `actor_type` | `text NOT NULL CHECK (actor_type IN ('bootstrap','user','system'))` | 当前为 `bootstrap`，未来 RBAC 为 `user` |
| `action` | `text NOT NULL CHECK (action <> '')` | 当前 `create_virtual_key`，未来 `update_quota` / `upsert_provider` 等 |
| `resource_type` | `text NOT NULL CHECK (resource_type <> '')` | 当前 `virtual_key` |
| `resource_id` | `text` | 当前写 `api_keys.id` 字符串；允许未来非 UUID 资源 |
| `"before"` | `jsonb` | 修改前快照；create 类动作为 `NULL` |
| `"after"` | `jsonb` | 修改后快照；严禁包含 virtual key secret / Authorization / prompt 全文 |
| `ip` | `inet` | 从 `RemoteAddr` 解析，解析失败写 `NULL` |
| `user_agent` | `text NOT NULL DEFAULT ''` | 只存 UA，不存敏感 header |
| `request_id` | `text NOT NULL DEFAULT ''` | `httpx.RequestIDFromContext`，不唯一；一个请求未来可能有多条审计 |
| `status_code` | `integer NOT NULL DEFAULT 0` | proposal 额外字段，用于区分成功写操作与失败写尝试 |
| `created_at` | `timestamptz NOT NULL DEFAULT now()` | UTC 由 PG 负责 |

`before` / `after` 选择 `JSONB`，不选 `TEXT`。理由：PG 17 对 JSONB 稳定；admin 写操作 QPS 很低，存储/写入开销可接受；T-014 审计页和 Phase 3 导出时可按字段过滤、脱敏、局部索引。实施时 SQL 中对 `"before"` / `"after"` 使用双引号，避免与 trigger 语法词混淆。`nil` 快照写 SQL `NULL`，空对象才写 `{}`，两者语义不同。

迁移策略：`ALTER TABLE audit_logs RENAME COLUMN actor TO actor_id`，新增 `actor_type` / `request_id` / `status_code`，`ip_address` 改名为 `ip`，`before_state` / `after_state` 改名为 `"before"` / `"after"`。down 反向恢复占位 schema。

## 2. 索引设计

保留现有 `audit_logs_created_at_idx`，新增两条 T-013 主索引：

- `audit_logs_actor_created_at_idx ON audit_logs (actor_id, created_at DESC)`
- `audit_logs_resource_created_at_idx ON audit_logs (resource_type, resource_id, created_at DESC)`

暂不新增 `action`、`status_code`、`request_id`、`user_agent`、JSONB GIN 索引。理由：Phase 2-A 只有一个写端点，索引过多会放大后续 quota/provider 高频写操作的写入成本；T-014 审计页先按 actor / resource / time 过滤即可。若 T-014 需要 request_id 精确检索，可单独补 `audit_logs_request_id_idx`；若 Phase 3 要按 `"after"->>'key_prefix'` 检索，再加表达式或 GIN 索引。

## 3. before 状态获取

推荐方案 A：handler 显式调用 `audit.SetBefore(ctx, snapshot)` / `audit.SetAfter(ctx, snapshot)`，middleware 只负责统一采集 actor/IP/UA/request_id/status 并异步 Record。

取舍：

- A 侵入 handler，但最准确，能复用 handler 已经查到的业务对象，不额外打 DB。
- B middleware 反查会让 audit 层知道每种 resource 的表结构，耦合高，还会给每个写请求增加至少一次 query。
- C 只记 after 最快，但 quota/provider/RBAC 修改会丢失关键审计证据，不适合作为 Phase 2-A 底座。

实现建议：`auditMiddleware` 在 `adminAuthMiddleware` 之后执行，给 request context 注入一个可变的 audit scope 指针；handler 通过 `audit.SetAction` / `audit.SetResource` / `audit.SetBefore` / `audit.SetAfter` 写入 scope。当前 `POST /api/admin/dev/virtual-keys` 在创建成功后设置 `action=create_virtual_key`、`resource_type=virtual_key`、`resource_id=<api_key_id>`、`before=nil`、`after={"api_key_id": "...", "key_prefix": "...", "organization_id": "...", "user_id": "..."}`，不写 `virtual_key` 明文。

## 4. 失败容忍策略

推荐 `WARN log + omt_audit_record_failures_total + 放行业务响应`，不因 audit 写失败返回 500。原因：admin 写操作已经被鉴权保护；audit 失败多半是 DB 短暂不可用或 schema/连接问题，返回 500 会让管理员反复重试，可能造成重复创建或更差的恢复体验。安全取舍是“鉴权失败闭门，审计失败高声告警但不阻断已授权操作”。

工程实现参考 `internal/usage`：handler 响应后用 goroutine + `context.WithTimeout(context.Background(), 2*time.Second)` 调 `Recorder.Record`；失败时 `logger.Warn("audit record failed", "request_id", ..., "action", ..., "resource_type", ..., "err", err)`，不记录 Authorization / secret / prompt。metric 用标准库 `expvar` 暴露 `omt_audit_record_failures_total`，不新增 Prometheus 依赖。

## 5. actor_id 占位与未来接入路径

`internal/audit` 不直接解析 token，也不依赖 admin RBAC 包。定义：

```go
type Actor struct {
    ID   string
    Type string
}

type ActorResolver func(*http.Request) Actor
```

T-013 的 `cmd/admin` 使用 `BootstrapActorResolver`：只要请求已通过 `adminAuthMiddleware`，actor 固定为 `{ID:"bootstrap", Type:"bootstrap"}`。这满足“从 Authorization header 解出 actor_id”在 bootstrap 阶段只能得到占位身份的现实约束，也避免重复比较 token。

T-005b 接入时只替换 resolver：从 `auth.SubjectFromContext` 或新的 admin subject context 取 user UUID 与角色，返回 `{ID:userID.String(), Type:"user"}`。`internal/audit.Recorder`、Postgres store、migration 均无需变化。

## 6. 测试方案

默认单测矩阵至少 8 case，覆盖 T-013 要求的 6 case：

- 写请求触发 audit，且 action/resource/actor/ip/ua/request_id/status 填充正确。
- GET `/api/admin/overview` 与 `/healthz` 不触发 audit。
- Recorder 返回错误时业务仍返回成功，日志为 WARN，`omt_audit_record_failures_total` 增加。
- DB 未配置时 `NewRecorder(nil, logger)` / no-op store 无 panic、业务放行。
- Postgres store SQL 字段断言，特别是 `actor_id`、`actor_type`、`"before"`、`"after"`、`request_id`。
- `before` / `after` JSONB 序列化：nil -> SQL NULL，map/struct -> JSON，包含 key_prefix 但不包含 secret。
- 当前 dev-vkey endpoint 成功创建后只写 key_prefix / api_key_id，不写明文 `virtual_key`。
- 失败业务请求仍可写一条 status_code >= 400 的审计尝试，若 action 缺失则使用 `unknown_admin_write`，避免写操作静默漏审。

`internal/audit` 覆盖率目标 ≥ 85%。默认测试不引入 testcontainers，避免新增依赖触发许可证 propose；Postgres store 用现有 fake SQL driver 模式做参数断言。真 PG 测试采用现有项目风格新增 `//go:build e2e`，读取 `OMNITOKEN_TEST_DATABASE_URL`，跑 `migrate up` 后插入/查询 `audit_logs`。这不是 mock DB；但默认 `go test ./...` 不消耗 Docker 或网络，符合当前仓库测试习惯。
