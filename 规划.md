文档版本: V1.1 (2026.05.11)
项目定位: 统一的 AI API 代理与管控网关，对内提供统一接口，对外屏蔽多模型差异，并实现细粒度的成本、权限与额度控制。

一、 系统架构设计

系统采用 API Gateway (API 网关) 模式。企业内部的应用和用户不直接请求 OpenAI 或 Anthropic，而是请求本系统。

请求链路:
企业应用/终端用户 -> OmniToken 网关 (鉴权、扣减额度) -> 大型语言模型 (GPT-4o/Claude/等)
大型语言模型 -> OmniToken 网关 (拦截流式响应、统计Token、异步落库) -> 企业应用/终端用户

架构分层

接入层 (Gateway): 处理 HTTP/SSE (Server-Sent Events) 请求，协议转换，统一采用 OpenAI API 格式作为企业内部标准接口。

拦截管控层 (Interceptor): 基于 Redis 的毫秒级鉴权、并发限流 (Rate Limiting)、可用额度检查。

核心路由层 (Router): 负载均衡、模型 Fallback（当A模型宕机自动切换B模型）、渠道池管理（管理多个官方或代理API Key）。

异步审计层 (Auditor): 利用消息队列处理请求日志、Token 计算和成本核算，不阻塞用户响应。

管理控制层 (Admin API): 为前端 Dashboard 提供 RESTful/GraphQL API。

二、 技术选型建议 (2026年主流)

为了应对高并发的流式转发和低延迟要求，推荐以下技术栈：

核心开发语言: Go (Golang)。Go 的协程 (goroutine) 天然适合处理海量 HTTP 代理和流式连接（SSE），且内存占用极低。

备选: Python (FastAPI/LiteLLM) 适合快速迭代，但高并发代理性能不如 Go；Node.js (NestJS) 适合全栈团队。

数据库 (RDBMS): PostgreSQL。用于存储用户数据、额度配置、模型费率表、渠道配置。

缓存与限流: Redis。这是系统的灵魂。用于存储高频验证的 API Key、实时扣减 Token 额度、RPM/TPM 限流。

消息队列: RabbitMQ 或 Kafka。网关在处理完请求后，将消耗记录丢入队列，由消费者异步写入数据库，保证网关极速响应。

Token 计算库: 使用各大模型官方提供的 Tokenizer (如 tiktoken) 的 Go 移植版本。

三、 核心模块详细设计

1. 统一 API 网关模块 (Core Proxy)

协议兼容: 强制对外暴露兼容 OpenAI API 的端点（如 /v1/chat/completions）。

协议转换: 当用户请求 claude-3-5-opus 时，网关在内部将 OpenAI 格式的 JSON 转换为 Anthropic 的结构，请求上游，再将返回结果转换回 OpenAI 格式给用户。

流式处理 (Streaming): 这是最难的部分。必须以 Reverse Proxy 方式透传 SSE 流，并在此过程中劫持并拼接内容，以便在流结束时准确计算消耗了多少 Token。

2. 鉴权与额度管控模块 (Auth & Quota)

双层校验:

网关 API Key: 用户/应用自带的内部 Token。

上游 API Key: 系统存储的真实大模型厂商 Token池。

高并发扣费 (Redis Lua 脚本): 请求到达时，使用 Lua 脚本在 Redis 中原子性地检查用户剩余额度。额度不足则直接拦截（返回 HTTP 402 Payment Required）。

预扣与结算机制: 流式请求无法预知最终输出多少 Token。机制应为：请求时预扣 max_tokens（或历史平均值），请求结束后，根据实际 Token 量进行多退少补。

3. 日志与计费中心 (Billing & Auditing)

精细化费率: 数据库需维护一张动态费率表。因为 GPT-4o 和 Llama-3 的价格不同，输入 (Prompt) 和输出 (Completion) 的单价也不同。

成本计算: Cost = (Prompt_Tokens * Prompt_Rate) + (Completion_Tokens * Completion_Rate)。

隐私合规: 提供全局开关，决定是否将用户的具体 Prompt 内容落盘。通常企业级应用为了安全，只记录 Token 消耗量、时间、模型，不保存对话明细（或进行脱敏处理）。

4. 前端控制台支撑 API (对接你看到的前端)

GET /api/admin/overview: 返回总消耗、活跃用户数、预估总额（供 KPI 卡片使用）。

GET /api/admin/charts/trend: 返回过去 30 天的时间序列数据。

GET /api/admin/users: 支持分页、搜索，返回用户列表及进度条所需的 used 和 quota 数据。

POST /api/admin/users/{id}/quota: 管理员修改用户额度的接口。

四、 数据库表结构草案 (PostgreSQL)

organizations / users: 组织架构与用户信息。

user_api_keys: 分配给内部用户的密钥（哈希存储）。

channels: 上游渠道配置（如 OpenAI 官方, Azure OpenAI, AWS Bedrock 等），包含真实 API Key（需加密存储）。

models: 模型字典与费率表（如 gpt-4o, 输入 5$/1M, 输出 15$/1M）。

usage_logs: 每次调用的明细。

字段: id, user_id, model, prompt_tokens, completion_tokens, total_cost, request_time, duration_ms。

user_quotas: 用户的额度表（周期性重置，如按月）。

五、 阶段性开发演进计划 (Milestones)

Phase 1: MVP (最小可行性产品) - 预计 2-3 周

目标: 让流量跑通，能防超刷。

实现基础的 HTTP/SSE 代理转发。

实现只针对单一模型（如 GPT-4o）的代理。

实现基于数据库的简单鉴权。

实现拦截响应中的 usage 字段并简单记录到数据库。

Phase 2: 高并发与计费引擎 - 预计 3-4 周

目标: 达到企业级生产可用。

引入 Redis，实现全内存的鉴权和额度扣减。

实现流式响应 (Stream) 的截获与精确 Token 计算（使用 Tiktoken）。

引入异步队列（RabbitMQ），将高频的日志写入操作异步化。

开发前端 Dashboard 所需的所有管理端 API，并与前端联调。

Phase 3: 路由、高可用与精细化管理 - 预计 3 周

目标: 降本增效，系统鲁棒性。

实现多模型协议统一（接入 Claude, Gemini 等）。

实现渠道池负载均衡（Round Robin）与故障自动重试（Fallback）。

实现用户部门级别的配额管理、超额邮件/飞书告警提醒。

六、 2026年最新技术挑战提示

多模态 (Multimodal) Token 计算: 现在的模型大量输入图片、音频。你的 Tokenizer 不仅要计算文本，还需要根据图片的像素分辨率、音频时长来精确计算并折算为 Token。

Prompt Caching: 2026年许多厂商（如 Anthropic, OpenAI）支持了上下文缓存。这部分命中的 Token 价格通常便宜一半，你的后端在解析 usage 账单时，需要额外处理 cached_tokens 字段以保证成本核算准确。

4. 后端数据模型补充建议

在现有 PostgreSQL 草案基础上，建议补充或细化以下表：

organizations: 企业、部门、项目的租户根。

projects: 项目/应用维度，用于把同一企业内不同业务的消耗拆开。

api_keys: 内部虚拟 Key。必须哈希存储 key_secret，只保留 key_prefix 用于检索；字段包含 owner_type、owner_id、expires_at、status、models_allowlist、rpm_limit、tpm_limit、monthly_budget、tags、last_used_at。

upstream_credentials: 上游真实 Key 池。字段包含 provider、base_url、encrypted_secret、region、priority、weight、status、health_state、last_error、quota_hint、metadata。真实 Key 必须使用 KMS 或 envelope encryption 加密。

routes: 路由策略。字段包含 match_model、match_project、target_provider、target_model、fallback_chain、retry_policy、timeout_ms、weight、canary_percent、config_version、status。

model_catalog: 模型字典。字段包含 canonical_model、provider_model、provider、context_window、max_output_tokens、modalities、supports_streaming、supports_tools、supports_prompt_cache、supports_reasoning、status。

model_pricing: 价格表。字段至少包含 input_rate、output_rate、cached_input_rate、cache_creation_rate、reasoning_rate、image_rate、audio_rate、currency、effective_from、source_url、updated_at。

usage_events: 原始用量事件。字段包含 request_id、trace_id、org_id、project_id、user_id、api_key_id、route_id、upstream_credential_id、model_requested、model_actual、provider、status_code、error_code、latency_ms、ttft_ms、streaming、cache_status、created_at。

usage_token_breakdown: Token 明细。字段包含 prompt_tokens、completion_tokens、reasoning_tokens、cached_tokens、cache_creation_tokens、cache_read_tokens、image_tokens、audio_input_seconds、audio_output_seconds、total_tokens。

cost_ledger: 成本账本。不要只依赖 usage_logs 即时计算，建议将每次请求结算后的金额写入账本，字段包含 cost_usd、list_cost_usd、discount_amount、billing_policy_version、settlement_status。

audit_logs: 管理员行为审计。企业版必须支持不可篡改导出或至少 append-only 存储。

alerts: 告警规则与触发记录。支持 budget、quota、error_rate、latency、provider_down、abnormal_key_usage。

5. API 设计补充

建议新增以下管理端 API：

GET /api/admin/api-keys: Key 列表，支持 owner、状态、模型、最后调用时间筛选。

POST /api/admin/api-keys: 创建虚拟 Key，返回明文 Key 仅一次。

POST /api/admin/api-keys/{id}/rotate: 轮换 Key。

POST /api/admin/api-keys/{id}/disable: 禁用 Key。

GET /api/admin/providers: 上游渠道/Key 池列表。

POST /api/admin/providers/{id}/health-check: 手动测试渠道可用性。

GET /api/admin/routes: 路由策略列表。

POST /api/admin/routes/{id}/publish: 发布策略版本。

POST /api/admin/routes/{id}/rollback: 回滚策略版本。

GET /api/admin/usage/summary: 按时间、组织、项目、用户、Key、模型、渠道聚合的用量和成本。

GET /api/admin/logs: 请求日志，默认只返回摘要和脱敏字段。

GET /api/admin/logs/{request_id}: 单次请求详情，需要更高权限。

GET /api/admin/models: 模型目录与价格。

POST /api/admin/models/sync: 从厂商或人工配置同步模型价格。

GET /api/admin/audit-logs: 管理员操作审计。

6. 架构优先级调整建议

MVP 不建议一开始追求完整多模型协议转换。优先顺序应为：

1. OpenAI-compatible 入口稳定，支持非流式和 SSE 流式透传。
2. 虚拟 Key 鉴权、哈希存储、模型白名单、状态开关。
3. Redis RPM/TPM 限流与预算预扣。
4. 统一 usage 解析与异步落库，先做到能和厂商后台账单对齐。
5. 上游渠道池、权重、健康检查、自动 Fallback。
6. 管理台补齐 Key、渠道、路由、日志、成本五个核心页面。
7. Guardrails、DLP、Prompt/Response 内容审计、Langfuse/OpenTelemetry 导出。

7. 重要风险提示

不要在前端或日志中展示完整 API Key。创建后只展示一次，之后只展示 key_prefix。

不要把 Prompt/Response 明文日志作为默认开启项。企业客户通常会要求“仅记录元数据”或“可配置脱敏/保留周期”。

不要硬编码模型名称和价格。2026 年模型迭代很快，前端图表里的 GPT-4o、Claude 3.5、Gemini 1.5 等只适合 Mock，正式系统应全部从 model_catalog 与 model_pricing 动态读取。

不要只按总 tokens 计费。OpenAI、Anthropic、Gemini、Bedrock 等都可能有不同的 cached、reasoning、tool、image、audio 计费字段，必须做 provider-specific usage mapper。

不要把“代理转发”和“管理控制台”写死在一个进程里。建议逐步拆分为 Gateway Data Plane、Admin API Control Plane、Async Worker、Scheduler/Health Checker 四类服务。

八、 技术栈最终选型 (Lock-in)

为了 Codex 实施时不再反复"选型讨论"，本节为强约束。如需变更，先在本节备案再改。

| 维度 | 选型 | 版本/备注 |
| --- | --- | --- |
| 后端语言 | Go | 1.23+（goroutine 处理 SSE 与上游代理） |
| HTTP 框架 | net/http + chi (github.com/go-chi/chi/v5) | 轻量、与标准库兼容、便于做 SSE 反向代理 |

| 数据库 | PostgreSQL 16 | 主存储 |

| 数据库迁移 | golang-migrate/migrate | SQL 文件迁移，禁用代码内嵌迁移 |
| 缓存/限流 | Redis 7 + Lua | 限流与额度预扣 |
| 消息队列 | NATS JetStream (MVP) → Kafka (规模化) | MVP 阶段 NATS 部署简单；Kafka 在 Phase 3 引入 |

十、 测试策略与质量门

强制规则:

1. 单元测试: `internal/` 下每个包覆盖率 ≥ 70%，核心包（quota, billing, proxy, auth）≥ 85%。
2. 集成测试: 使用 testcontainers-go 启动真实 Postgres / Redis / NATS，禁止 mock 数据库。
3. 协议兼容测试: 维护"黄金请求/响应"语料库（`testdata/golden/`），任何 SSE/协议转换改动都要回归。
4. 负载基线: Phase 2 验收前提交一次 `vegeta` 报告（4 vCPU、单实例、1000 RPS 非流式 / 200 并发 SSE）。
5. 端到端: Playwright 脚本覆盖管理台核心 5 个页面（Phase 2 末）。

CI 必过项: `go vet ./...`、`golangci-lint run`、`go test -race ./...`、`docker build`、`migrations up && down` 双向验证。

十一、 安全与合规基线

入网最小集（Phase 0 末必须达到）:

1. 所有上游 API Key 落库前以 AES-256-GCM 加密；主密钥仅从环境变量或 KMS 加载，绝不进入仓库。
2. 内部虚拟 Key 仅存 `prefix(8 位) + sha256(secret)`，明文仅在创建时返回一次。
3. 默认不落 Prompt/Response 明文，配置项 `LOG_BODY_MODE` 取值 `off | hashed | full`，默认 `off`。
4. 所有管理端写操作必须写 `audit_logs`，含 actor、IP、user-agent、before/after。
5. SSE 转发禁止把上游错误堆栈透传给终端用户，统一 502/503 包装。
6. 日志、错误信息、metric 标签均不允许出现完整 Key 或 Authorization Header。

十二、 修订后的阶段计划 (Phase 0 - Phase 3)

Phase 0: 项目初始化 (3-5 天，Codex 主导，Claude 校验)
- 仓库脚手架：上方第九节目录结构落地，加入 Makefile 与 `make help`。
- `deploy/docker-compose.yml` 一键起 pg/redis/nats，含 seed 脚本。
- `cmd/gateway` 给出最小 `/healthz`、`/v1/models`（返回静态列表）。
- `cmd/admin` 给出 `/api/admin/overview` Mock，能被现有 `测试前端.html` 通过 fetch 调通。
- GitHub Actions 跑 lint + test + docker build。
- 首批 ADR：`0001-tech-stack.md`、`0002-monorepo-layout.md`。
- 验收: 新人 clone 后 30 分钟内 `make up && curl localhost:8080/healthz` 通过。

Phase 1: MVP 代理与计费 (2-3 周)
- OpenAI 兼容 `/v1/chat/completions` 非流式 + SSE 流式透传。
- 虚拟 Key 校验 + 模型白名单。
- Redis Lua 预扣 + 实际 usage 多退少补。
- usage 解析支持 OpenAI 原生字段 (含 `prompt_tokens_details.cached_tokens`)。
- NATS 异步落 `usage_events` + `cost_ledger`。
- 验收: 与 OpenAI 官方账单误差 ≤ 1%；1000 RPS 非流式 P95 < 80ms（代理本身开销）。

Phase 2: 多渠道、协议转换、管理台 (3-4 周)
- 接入 Anthropic / Gemini / Azure OpenAI，统一 usage mapper（含 cache_creation/cache_read、reasoning）。
- 上游渠道池：权重、优先级、健康检查、Fallback。
- 管理端 API 完整版（见原文第三节 4 与第五节 5）。
- 正式前端工程化（Next.js）启动，覆盖 Keys / Providers / Routes / Usage / Logs 五页。
- 验收: 管理台五页可视化对齐 LiteLLM / Portkey 功能基线。

Phase 3: 治理、可观测、生产化 (3 周)
- Guardrails（PII / 敏感词 / Prompt Injection 检测，插件化）。
- 预算/错误率/延迟告警，Webhook + 飞书/邮件。
- OpenTelemetry 导出 + Langfuse 集成。
- Data Plane / Control Plane / Worker / Scheduler 拆分为独立部署单元。
- 验收: 通过一次内部红队演练 + 7 天 SLO 99.9% 演示。

十三、 协作机制 (Claude × Codex)

完整规则见 `CLAUDE.md` 与 `AGENTS.md`，本节仅放交接物：

- `TASKS.md`: Claude 拆分任务、写入接受标准；Codex 自上而下取任务，完成后回写状态。
- `REVIEW.md`: Codex 提交后，Claude 写入 Review 结论（CRITICAL / HIGH / MEDIUM / NIT）。Codex 负责 CRITICAL/HIGH 全部修复，MEDIUM 视情况。
- 不允许 Codex 在未更新 `TASKS.md` 的情况下私自开新工作；不允许 Claude 在未读取 Codex 最近提交的情况下下达新任务。
- 规划变更（包括第八节技术栈）必须先在 `规划.md` 改动并打上 `[v1.x]` 标记，再生效。
